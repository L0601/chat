# 当前 App UI 设计文档

## 1. 文档目的

这份文档用于说明当前 LunaDesk App 的移动端 UI 设计方向、页面结构、交互原则和实现约束，方便后续继续迭代聊天页、设置页，以及新增页面时保持统一风格。

当前文档对应的是“现有 app”，不是未来的独立业务页设计稿。

## 2. 当前页面范围

当前 App 主要包含两个页面：

- 聊天页
- 设置页

当前导航方式：

- 默认进入聊天页
- 左上角三横按钮打开侧边栏
- 侧边栏内进入设置页
- 侧边栏和设置页都可执行“重置当前聊天上下文”

## 3. 设计目标

- 界面尽量接近 ChatGPT 手机端的单页聊天体验。
- 默认视觉中心始终放在“对话内容”上，而不是操作按钮上。
- 顶部和底部控件尽量轻量，不抢占聊天区空间。
- 键盘弹出时，聊天内容仍然要可见，不能出现输入框把消息整体挤没的问题。
- 保持浅色、圆角、轻悬浮的视觉语言，避免厚重卡片堆叠感。

## 4. 整体视觉基线

### 4.1 页面风格

- 使用浅色渐变背景。
- 内容容器以近白色卡片为主。
- 圆角普遍保持在 18dp 到 30dp 之间。
- 控件边框弱化，更多依赖留白、色块和轻阴影做层级。

### 4.2 色彩方向

- 背景采用米白到浅蓝绿的柔和渐变。
- 主要文字使用偏深灰绿。
- 状态提示使用蓝色或暖色，避免全页面高饱和。
- 用户消息和助手消息用不同浅色块区分，但都保持低刺激度。

### 4.3 间距规范

- 页面左右主边距控制在 12dp 左右。
- 卡片之间垂直间距保持 8dp 到 10dp。
- 顶部按钮、输入栏、提示条之间保持轻量间隔，不要堆得太紧。

## 5. 页面结构说明

### 5.1 聊天页

聊天页是当前默认首页，也是主使用场景。

结构分为四部分：

#### 顶部栏

- 左侧是三横菜单按钮。
- 中间是轻量状态胶囊，例如连接状态、准备开始新对话等。
- 右侧是消息计数的小按钮位。

设计原则：

- 顶部栏要轻，不做厚重标题区。
- 顶部控件不占用过高空间。
- 用户进入页面后，第一关注点应该是聊天内容，而不是顶部导航。

#### 提示条

- 用于展示“正在生成”“已停止生成”“已重置本轮对话”等内联提示。
- 位置在顶部栏和消息区之间。

设计原则：

- 提示条高度尽量低。
- 能关闭，避免长期占位。

#### 消息区

- 消息区是聊天页主体，使用最大剩余空间。
- 无消息时显示空状态文案。
- 有消息时展示用户消息和助手消息列表。

设计原则：

- 消息区始终优先占空间。
- 键盘弹出时，消息区缩小但不能消失。
- 新消息和流式回复时自动滚动到底部。

#### 底部输入区

- 左侧是轻量扩展按钮位。
- 中间是输入框。
- 右侧是发送/停止按钮。

设计原则：

- 输入区高度尽量压缩。
- 输入框默认单行优先，只在必要时有限扩高。
- 不把“重置”“设置”等低频操作塞进输入区。
- 输入区只负责输入，不承担过多业务入口。

### 5.2 设置页

设置页承载当前的配置能力，不再和聊天页通过顶部 tab 并列切换。

结构分为四部分：

#### 顶部栏

- 左侧返回按钮。
- 中间是“设置”标题。
- 右侧保留对称占位，保证视觉平衡。

#### 当前会话卡片

- 提供“重置上下文”操作。
- 用于清空当前聊天记录和输入框内容。

#### 配置卡片

- 服务地址
- 温度
- 最大输出
- 保存
- 测试连接
- 拉取模型

#### 模型列表卡片

- 展示模型列表
- 支持当前模型标识
- 支持切换中状态

设计原则：

- 设置页仍然保持轻量，不做复杂分组导航。
- 配置操作和模型操作保持卡片分区，避免混在一起。

## 6. 导航设计说明

当前导航采用“聊天为主，设置为辅”的结构。

设计原因：

- 聊天是主场景，不应该和设置平权占据顶部。
- 顶部双 tab 会直接挤压聊天可视区域。
- 侧边栏更适合放设置、重置等低频操作。

当前侧边栏包含：

- 聊天
- 设置
- 重置当前聊天

后续如果增加页面，也建议优先继续挂在侧边栏，而不是把顶部改回 tab 切页。

## 7. 键盘与输入交互原则

这是当前 UI 改造的重点。

需要满足：

- 键盘弹出时，输入区贴底抬起。
- 消息区保留可见内容。
- 不允许整页被 `imePadding` 一起顶上去。
- 输入完成后，页面视觉中心仍然是对话，而不是只剩输入框。

实现原则：

- 键盘适配尽量集中在底部输入区。
- 消息列表使用剩余空间布局。
- 页面主体不要因为键盘弹出而整体垂直居中错位。

## 8. 空状态设计

当前无消息时，聊天页显示欢迎式空状态。

设计原则：

- 空状态和正常聊天态保持同一套页面骨架。
- 只替换消息区内容，不切换成另一种页面结构。
- 文案简洁，明确引导用户从侧边栏进入设置。

## 9. 组件设计原则

后续继续补 UI 时，建议遵循这些约束：

- 单个组件职责尽量单一。
- 容器组件负责布局，业务组件负责显示，不混杂过重逻辑。
- 输入栏、顶部栏、提示条、空状态、消息气泡都可以继续抽成通用组件。
- 样式统一优先于局部花哨。

## 10. 后续页面扩展建议

如果后面继续增加页面，建议依然基于当前设计基线扩展：

- 页面默认保留浅色渐变背景
- 页面主内容使用圆角卡片容器
- 顶部栏保持轻量
- 高频操作保留在主视区或底部
- 低频操作优先放侧边栏或二级页

适合后续新增的页面包括：

- 历史会话页
- 价格/行情相关页
- 关于页
- 通用设置页

## 12. 总结

当前 app 的 UI 核心思路可以概括为：

- 聊天优先
- 设置后置
- 输入轻量
- 内容优先
- 键盘友好

后面不管是继续抠聊天细节，还是加新页面，都建议沿着这条线扩展，不要再回到“顶部塞很多按钮、底部堆很多操作、内容区被挤压”的结构。
